## 3. 核心模块设计

### 3.1 RoomManager（房间管理器）

```javascript
class RoomManager {
  rooms: Map<roomId, Room>
  
  // 核心方法
  createRoom(roomId, width, height) → Room
  getRoom(roomId) → Room | undefined
  removeRoom(roomId) → void
  addPlayer(roomId, socketId, playerName) → Player
  removePlayer(roomId, socketId) → void
  getRoomList() → RoomInfo[]
}
```

**职责：**
- 管理所有游戏房间的创建、查询、销毁
- 处理玩家的进出房间逻辑
- 维护房间列表，支持大厅展示

**房间生命周期：**
```
创建 → 玩家加入 → 游戏进行 → 玩家离开 → (空房间销毁)
                          ↓
                    持久房间保留（如"公共农场"）
```

### 3.2 FarmGame（农场游戏逻辑）

```javascript
class FarmGame {
  width: number          // 农场宽度
  height: number         // 农场高度
  plots: Plot[][]        // 地块二维数组
  players: Map<id, Player>  // 玩家数据
  gameStatus: string     // 游戏状态
  startTime: number      // 开始时间
  
  // 核心方法
  addPlayer(socketId, name) → Player
  removePlayer(socketId) → void
  movePlayer(socketId, x, y) → Result
  plant(socketId, cropType) → Result
  water(socketId) → Result
  harvest(socketId) → Result
  getState() → GameState
}
```

**职责：**
- 管理单个房间的游戏状态
- 处理玩家操作（移动、种植、浇水、收获）
- 运行生长更新循环（每秒一次）
- 提供完整游戏状态快照

### 3.3 Plot（地块）

```javascript
class Plot {
  x, y: number           // 坐标
  soilMoisture: number   // 土壤湿度 0-100
  crop: string | null    // 作物类型
  plantedAt: number      // 种植时间戳
  growthStage: number    // 0-3 生长阶段
  isWatered: boolean     // 当日是否浇水
  owner: string | null   // 种植者名称
  
  // 核心方法
  plant(cropType, playerName) → Result
  water() → Result
  harvest() → Result
  updateGrowth() → void
  getState() → PlotState
}
```

**职责：**
- 管理单个地块的状态
- 计算作物生长阶段
- 处理种植/浇水/收获操作

### 3.4 游戏循环（生长系统）

```
┌─────────────────────────────────────┐
│       setInterval(1000ms)           │
└───────────────┬─────────────────────┘
                ↓
        ┌───────────────┐
        │ 遍历所有地块   │
        └───────┬───────┘
                ↓
        ┌───────────────┐
        │ updateGrowth()│
        └───────┬───────┘
                ↓
    ┌───────────────────────────┐
    │ 计算已生长时间            │
    │ 浇水状态加速 1.5x         │
    │ 更新 growthStage (0-3)    │
    └───────────────────────────┘
```

## 4. 数据流

### 4.1 玩家操作流

```
┌─────────┐     ┌──────────┐     ┌─────────┐     ┌──────────┐
│ 客户端   │────→│ WebSocket│────→│ 游戏逻辑 │────→│ 状态广播  │
│ 触发操作 │     │ 发送事件  │     │ 验证执行 │     │ 全员更新  │
└─────────┘     └──────────┘     └─────────┘     └──────────┘
     ↑                                              │
     └──────────────────────────────────────────────┘
                         接收 game-state
```

### 4.2 状态同步策略

| 同步方式 | 使用场景 | 说明 |
|---------|---------|------|
| 全量广播 | game-state | 每次操作后广播完整状态 |
| 增量更新 | （预留） | V2.0 优化，只传变化 |
| 乐观更新 | 客户端移动 | 本地先响应，服务端确认 |

## 5. 状态管理

### 5.1 服务端权威模式

```
所有游戏逻辑计算在服务端完成
    ↓
客户端发送操作请求
    ↓
服务端验证并执行
    ↓
广播结果给所有客户端
```

### 5.2 客户端预测（V1.0 简化）

当前版本：服务端确认后才更新显示
V2.0 规划：
- 移动操作本地先执行
- 服务端返回实际位置
- 如有偏差平滑修正

### 5.3 冲突处理

| 冲突场景 | 处理策略 |
|---------|---------|
| 两玩家同时种一块地 | 先到先执行，后者失败 |
| 移动超出边界 | 服务端拒绝，保持原位 |
| 资金不足 | 服务端拒绝，提示失败 |
| 收获未成熟作物 | 服务端拒绝 |

## 7. 性能优化

### 7.1 当前优化

- **生长计算**：每秒批量更新，非每个客户端请求
- **房间隔离**：不同房间状态完全隔离
- **空房清理**：非持久房间自动销毁

### 7.2 V2.0 优化方向

- **增量同步**：只传输变化的地块和玩家
- **空间分区**：视野外玩家/地块停止同步
- **快照压缩**：二进制协议替代 JSON

## 10. 关键设计决策

### 10.1 为什么选 Socket.io？

| 优势 | 说明 |
|------|------|
| 自动重连 | 网络波动时自动恢复连接 |
| 房间机制 | 内置 room 概念，天然匹配游戏房间 |
| 事件驱动 | 声明式事件处理，代码清晰 |
| 降级支持 | 自动降级到 HTTP 长轮询 |

### 10.2 为什么服务端权威？

- **安全性**：防止客户端作弊
- **一致性**：所有玩家看到相同状态
- **简化客户端**：客户端只负责渲染和输入

### 10.3 为什么每秒更新生长？

- **平衡性能与实时性**：每秒足够感知生长变化
- **减少计算量**：避免高频计算
- **批量处理**：一次性更新所有地块
